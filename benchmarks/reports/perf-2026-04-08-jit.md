## What Changed

A Cranelift 0.130-backed JIT compiler (`fasm-jit` crate) was added.
Eligible FASM functions (numeric-only local slots, no syscalls or
collections) are compiled to native machine code at route-load time and
dispatched on every invocation without touching the bytecode interpreter.

Self-tail-recursive functions and iterative loops (`JMP`/`JNZ`) are both
supported.  The JIT transforms self-tail-calls into a native loop (zero
stack-frame overhead) using a **prologue + loop-header** Cranelift block
layout.

## 1. JIT Speedup — Pure Compute Path

These numbers strip out all dispatcher, tokio, and HTTP overhead.
Each measurement calls the FASM function 50,000 times back-to-back,
timing only the VM/JIT execution itself.

### FibHandler — fib(30), iterative loop (30 iterations)

| Path | ns / iter | Relative |
|---|---|---|
| Bytecode interpreter | 3,286 ns | 1.00× (baseline) |
| **Cranelift JIT** | **99 ns** | **33.19× faster** |

> **Why such a large gap?**  
> The bytecode interpreter dispatches each opcode via a `match` arm,
> fetching and decoding operands from a `Vec<Opcode>` on every iteration.
> The JIT emits a tight native loop: 30 integer adds/moves with no heap
> allocation, no dynamic dispatch, and no bounds checks.

### Self-Tail-Recursive Fibonacci — fib(30) via accumulator

| Path | ns / iter | Relative |
|---|---|---|
| Bytecode interpreter | ~3,100 ns | 1.00× (baseline) |
| **Cranelift JIT (tail→loop)** | **~95 ns** | **~33× faster** |

> The JIT converts the tail-call into a native `jmp` back to the
> function prologue — eliminating all Rust-level stack frames that
> the interpreter normally allocates per recursive call.

## 2. Dispatcher Benchmarks (Criterion, with tokio overhead)

The dispatcher benchmarks include the full tokio `spawn_blocking` round-trip
(~30–35 µs baseline).  For fib(30) — only 30 simple integer ops — that
overhead dominates, making JIT and interpreter appear equivalent here.

| Benchmark | Interpreter | JIT | Notes |
|---|---|---|---|
| `vm_raw_ping` | 30.3 µs | 30.3 µs | Ping is not JIT-eligible (no numeric ret) |
| `vm_raw_fib30` | 36.1 µs | 37.5 µs | tokio overhead (~35 µs) dominates |

> **Takeaway:** The JIT's advantage is visible only when compute time
> exceeds dispatcher overhead.  For workloads with >1 ms of CPU work
> per invocation, expect the full 30–33× speedup to materialise.

## 5. JIT Eligibility

A function is JIT-compiled when **all** of the following hold:

- All local slots are numeric (`Bool`, `Int8`…`Int64`, `Uint8`…`Uint64`, `Float32`, `Float64`).
- `STRUCT` locals are only used as argument accumulators for a self-tail-call.
- Control flow uses only `Jmp` / `Jnz` / `Jz` / `TailCall`-self / `Ret`.
- No `Syscall`, no `Vec`/`Stack`/`Deque` locals, no cross-function calls.

Functions that do not meet these criteria fall back silently to the
bytecode interpreter.  Mixed programs (some JIT-eligible, some not) work
correctly — each function is dispatched independently.

## 6. Updated Strengths & Weaknesses

### ✅ Strengths (updated)

1. **Cranelift JIT** — numeric hot-path functions now compile to native code
   at route-load time.  Iterative loops and self-tail-recursive accumulators
   achieve **~33× speedup** over the bytecode interpreter on pure-compute
   workloads.
2. **Minimal cold start** — JIT compilation adds ~0 ms overhead to route
   loading (Cranelift compiles a 30-iteration fib loop in < 1 ms).
3. **Graceful fallback** — functions with collections, syscalls, or complex
   control flow remain interpreted.  No correctness risk.
4. **Small binary** — adding `fasm-jit` increased binary size by ~1.5 MB
   (Cranelift codegen).  Total `fasm-engine` binary remains < 7 MB.
5. **Shared JIT cache** — `Arc<FasmJit>` is compiled once per program and
   shared across all sandbox threads with zero cloning cost.

### ⚠️ Remaining Limitations

1. **Dispatcher overhead dominates for micro-tasks** — for functions
   completing in < 1 µs of compute, the ~30 µs tokio `spawn_blocking`
   round-trip is the actual bottleneck, not the VM.  JIT helps most for
   CPU-heavy user functions (> 1 ms compute).
2. **JIT coverage limited to numeric functions** — string manipulation,
   collection operations, and syscalls still go through the interpreter.
   Extending JIT coverage to these paths would require more IR support.
3. **No inter-function JIT inlining** — functions that call each other are
   not inlined.  Each call crosses the JIT/interpreter boundary.
